## Overview

The Documentation.AI REST API provides programmatic access to your documentation project through a standard REST interface. You can read pages, search content, manage branches, and push file changes — the same operations available in the dashboard, accessible from your own scripts, CI/CD pipelines, or backend services.

The Documentation.AI REST API and the [Authoring MCP Server](/ai/authoring-mcp-server) cover largely the same documentation-management actions through different interfaces. Use the REST API when you need direct HTTP access from scripts, backends, or CI/CD, and use the Authoring MCP Server when you are working from an MCP-compatible AI client or agent.

## Prerequisites

Before using the Documentation.AI REST API, make sure you have the following:

1. **A Documentation.AI account** on the Standard plan or above
2. **A published documentation project** with a connected repository
3. **Admin access** to the organization that owns the documentation project (required to create API keys)

## Getting Your API Key

1. Open the [Documentation.AI Dashboard](https://dashboard.documentation.ai)
2. Select the documentation project you want to access
3. Go to **Settings → API Keys**
4. Click **Create Key**
5. Choose a name, role (Viewer, Editor, or Admin), and expiration
6. Copy the generated key immediately — it will only be shown once

<Callout kind="alert">
  Store your API key securely. Do not commit it to version control or expose it in client-side code. Use environment variables or a secrets manager.
</Callout>

<Callout kind="info">

If you want AI-assisted authoring instead of direct HTTP requests, use the [Authoring MCP Server](/ai/authoring-mcp-server). This page covers the REST API.

</Callout>

## Base URL

```
https://api.documentationai.app/api/v1
```

## Authentication

All requests require an API key passed as a Bearer token in the `Authorization` header.

```bash
curl https://api.documentationai.app/api/v1/pages \
  -H "Authorization: Bearer dai_example_9xmk7q2r"
```

API keys are generated from the dashboard under **Settings → API Keys**. Each key is scoped to a single documentation project and uses the role selected when the key was created.

## Roles and Permissions

Every API key carries a role that determines which endpoints are accessible.

| Role | Read endpoints | Write endpoints |
| --- | --- | --- |
| **Viewer** | All read endpoints | None |
| **Editor** | All read endpoints | Push, branches, merge |
| **Admin** | All read endpoints | All write endpoints |

Write endpoints return `403 Forbidden` if the API key role does not have sufficient permissions.

## Rate Limiting

REST API requests are rate limited per API key. Unless a different custom limit is configured for the key, the default limit is **60 requests per minute**.

Rate limit information is included in responses via headers:

| Header | Description |
| --- | --- |
| `x-ratelimit-limit` | Maximum requests per minute |
| `x-ratelimit-remaining` | Requests remaining in the current window |
| `x-ratelimit-reset` | Unix timestamp when the window resets |
| `retry-after` | Seconds to wait before retrying. Present only on `429 Too Many Requests` responses. |

When the limit is exceeded, the API returns `429 Too Many Requests`. Wait for the number of seconds in the `retry-after` header before retrying the request.

## Quick Example

List all pages in your documentation:

<CodeGroup tabs="cURL,Node.js,Python">
```bash
curl https://api.documentationai.app/api/v1/pages \
  -H "Authorization: Bearer dai_example_9xmk7q2r"
```

```javascript
const response = await fetch('https://api.documentationai.app/api/v1/pages', {
  headers: { 'Authorization': 'Bearer dai_example_9xmk7q2r' }
});
const data = await response.json();
console.log(data.pages);
```

```python
import requests

response = requests.get(
    'https://api.documentationai.app/api/v1/pages',
    headers={'Authorization': 'Bearer dai_example_9xmk7q2r'}
)
print(response.json()['pages'])
```
</CodeGroup>

## Push Changes

Create, update, or delete files in a single commit:

```bash
curl -X POST https://api.documentationai.app/api/v1/push \
  -H "Authorization: Bearer dai_example_9xmk7q2r" \
  -H "Content-Type: application/json" \
  -d '{
    "commitMessage": "Update getting started guide",
    "operations": [
      {
        "type": "update",
        "path": "getting-started/quickstart.mdx",
        "content": "---\ntitle: Quickstart\n---\n\nUpdated content here."
      }
    ]
  }'
```

<Callout kind="info">
  Pushing to the deployment branch saves changes to the branch that powers the live site. Use a feature branch for draft changes, then merge when ready.
</Callout>

## Merge behavior

Branch merges affect deployment based on the target branch:

- **Merging into the deployment branch** triggers a live deployment after the merge succeeds.
- **Merging into a non-deployment branch** updates or triggers a preview deployment for that branch when preview sites are available.

Use a non-deployment branch when you want to review changes before they reach the live site. Merge into the deployment branch only when the changes are ready for visitors.
