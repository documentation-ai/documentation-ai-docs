## Overview

Configure interactive API testing environments that allow developers to make live API calls directly from your documentation. The playground automatically generates request forms from OpenAPI specifications and provides real-time response previews.

Use interactive playgrounds when you want to:

- Allow developers to test endpoints without external tools

- Provide immediate feedback on API responses

- Demonstrate authentication flows interactively

- Enable parameter experimentation within documentation

- Reduce friction in API adoption and testing

## Basic Configuration

### Enable Playground Features

Interactive playgrounds are automatically enabled when you include OpenAPI specifications in your documentation. The playground appears in the right sidebar alongside `Request` and `Response` components.

```json
{
  "navigation": {
    "groups": [
      {
        "group": "API Reference",
        "openapi": "api/openapi.yaml",
        "pages": [
          {
            "title": "User Management",
            "path": "api/users"
          }
        ]
      }
    ]
  }
}
```

### Playground Components

The interactive playground automatically generates:

- **Server controls** for choosing an OpenAPI server, filling server variables, or overriding the base URL

- **Parameter forms** for path, query, header, and body parameters

- **Named example pickers** when a parameter defines multiple OpenAPI examples

- **Authentication inputs** based on security schemes

- **Request preview** showing the actual HTTP request

- **Response viewer** with syntax highlighting and formatting

- **Status indicators** for successful and failed requests

## Authentication Configuration

### API Key Authentication

For APIs using API key authentication, the playground provides secure input fields:

```yaml
# In your OpenAPI specification
security:
  - ApiKeyAuth: []

components:
  securitySchemes:
    ApiKeyAuth:
      type: apiKey
      in: header
      name: X-API-Key
```

The playground renders:

- **Secure input field** for the API key

- **Header preview** showing how the key will be sent

- **Test button** to validate authentication

### Bearer Token Authentication

Configure Bearer token authentication for JWT-based APIs:

```yaml
security:
  - BearerAuth: []

components:
  securitySchemes:
    BearerAuth:
      type: http
      scheme: bearer
      bearerFormat: JWT
```

Features include:

- **Token input field** with secure handling

- **Automatic header injection** (`Authorization: Bearer <token>`)

- **Token validation** indicators

### OAuth 2.0 Integration

For OAuth 2.0 flows, the playground provides comprehensive authentication support:

```yaml
security:
  - OAuth2: [read, write]

components:
  securitySchemes:
    OAuth2:
      type: oauth2
      flows:
        authorizationCode:
          authorizationUrl: https://api.example.com/oauth/authorize
          tokenUrl: https://api.example.com/oauth/token
          scopes:
            read: Read access to resources
            write: Write access to resources
```

The playground includes:

- **OAuth flow initiation** with proper redirect handling

- **Scope selection** interface

- **Token management** with refresh capabilities

- **Automatic token injection** in requests

## Request Customization

### Parameter Forms

The playground automatically generates forms for all OpenAPI parameters:

<Request show-lines="true" tabs={[]}>
  ```javascript
  // For an endpoint: GET /users/{id}?include=profile
  const response = await fetch('/api/users/123?include=profile', {
    headers: { 'Authorization': 'Bearer TOKEN' }
  });
  ```
</Request>

Generated form includes:

- **Path parameter input** for `id`

- **Query parameter input** for `include`

- **Header management** for authentication and custom headers

- **Example values** from `example`, `examples`, and schema examples in your OpenAPI spec

- **Named example picker** when a parameter has multiple examples

- **Validation** based on OpenAPI schema constraints

When a parameter has one example, the playground can prefill or suggest that value. When it has multiple named examples, the input shows a dropdown so readers can choose a realistic value without treating the examples as an enum.

### Request Body Editing

For endpoints accepting request bodies, the playground provides:

```yaml
paths:
  /users:
    post:
      requestBody:
        content:
          application/json:
            schema:
              type: object
              properties:
                name:
                  type: string
                email:
                  type: string
                  format: email
              required: [name, email]
```

Features include:

- **JSON editor** with syntax highlighting

- **Schema validation** with real-time feedback

- **Example generation** from OpenAPI examples

- **Format validation** for email, date, etc.

## Advanced Features

### Server selection

The playground reads the `servers` array from your OpenAPI specification and shows a **Server** section above the request inputs. If your spec defines multiple servers, readers can choose the server they want to call from a dropdown.

```yaml
servers:
  - url: https://api.acme.com/{version}
    description: Production
    variables:
      version:
        default: v2
        enum:
          - v1
          - v2
  - url: https://sandbox.acme.com/{version}
    description: Sandbox
    variables:
      version:
        default: v2
```

The playground uses server variables to render editable inputs. Variables with `enum` values appear as dropdowns, and variables without `enum` values appear as text inputs. If a server URL includes a placeholder such as `{region}` but the variable is not declared, the playground still shows an input so the reader can fill the missing value.

<Callout kind="info" collapsed="false">
  A reader's selected server, variable values, and base URL override are saved in browser storage for the current server list. If you change the OpenAPI `servers` array, the saved selection resets safely to the spec defaults.
</Callout>

### Base URL override

Readers can override the resolved server URL directly in the **Server** section. This is useful when they want to test a staging API, a tenant-specific host, or a temporary preview environment that is not listed in the OpenAPI spec.

When a reader types a custom base URL:

- The playground uses the custom URL instead of the selected OpenAPI server.

- Server variable inputs are hidden because the custom URL is treated as the complete base URL.

- The reset button restores the selected OpenAPI server and its default variable values.

### Request History

The playground maintains request history for developer convenience:

- **Recent requests** with quick replay functionality

- **Saved requests** for frequently used API calls

- **Request bookmarking** with custom names and descriptions

- **History search** to find previous API interactions

## Security and Privacy

### Authentication Security

The playground implements secure authentication handling:

<ParamField path="tokenStorage" param-type="string" required="false" deprecated="false">
  Tokens are stored locally in browser storage, never transmitted to Documentation.AI servers.
</ParamField>

<ParamField path="httpsOnly" param-type="boolean" required="false" deprecated="false">
  All API requests use HTTPS encryption for secure data transmission.
</ParamField>

<ParamField path="tokenExpiry" param-type="boolean" required="false" deprecated="false">
  Automatic token cleanup on session end to prevent unauthorized access.
</ParamField>

### Proxy behavior

Playground requests are sent through Documentation.AI's server-side playground proxy instead of directly from the reader's browser. The proxy validates the target URL, forwards the request, and returns the upstream response to the playground.

The proxy blocks targets that resolve to internal network addresses. Requests to `localhost`, private IP ranges, link-local addresses, `.local` hosts, and `.internal` hosts fail before they reach the target API. This protects documentation visitors and infrastructure from server-side request forgery.

<Callout kind="alert" collapsed="false">
  APIs that only run on a developer laptop, private VPC hostname, or unresolved internal DNS name cannot be tested from the hosted playground. Use a public staging URL, a tunnel, or the base URL override with a reachable host.
</Callout>

## Troubleshooting

### Common Issues

<Callout kind="alert" collapsed="false">
  **CORS Errors**: If requests fail with CORS errors, ensure your API includes Documentation.AI domains in allowed origins.
</Callout>

<Callout kind="info" collapsed="false">
  **Authentication Failures**: Verify that authentication tokens are valid and have appropriate permissions for the endpoints being tested.
</Callout>

### Playground Not Appearing

If the interactive playground doesn't appear:

- **Verify OpenAPI specification** is properly referenced in `documentation.json`

- **Check file paths** to ensure OpenAPI files are accessible

- **Validate OpenAPI format** using tools like Swagger Editor

- **Ensure endpoints** have proper `operationId` values

### Request Failures

For failed API requests in the playground:

- **Check the Server section** to confirm the selected OpenAPI server, server variables, and base URL override resolve to the API you expect.

- **Use a reachable public host** if the response says requests to internal addresses are not allowed.

- **Verify DNS** if the response says the target host cannot be resolved.

- **Retry or shorten the request** if the response reports a timeout. Playground requests have a fixed timeout for long-running upstream calls.

- **Verify authentication** credentials and permissions.

- **Validate request parameters** against OpenAPI schema requirements.

### Performance Optimization

Optimize playground performance for better user experience:

- **Minimize OpenAPI file size** by removing unused definitions

- **Use response examples** to provide immediate feedback

- **Configure appropriate timeouts** for long-running endpoints

- **Implement request caching** where appropriate
