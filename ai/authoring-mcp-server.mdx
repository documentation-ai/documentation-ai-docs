## Overview

Use the Authoring MCP Server to let an MCP-compatible AI client work directly in your documentation project. It provides read and write access for editing pages, updating `documentation.json`, managing branches, reviewing changes, and deploying through your normal documentation workflow.

The Authoring MCP Server and the Documentation.AI REST API cover largely the same documentation-management tasks, but they expose them through different interfaces. Choose the Authoring MCP Server when you want an MCP-compatible AI client or agent to do the work for you, and choose the REST API when you want direct HTTP access from your own scripts, services, or CI/CD. For endpoint details, see the [API reference overview](/api-reference/overview).

<Callout kind="info">

Use this server for authoring and documentation operations through an MCP client. If you need direct REST API access instead, use the Documentation.AI REST API. If your AI tool only needs read-only access to published content, use the [Docs MCP Server](/ai/docs-mcp-server).

</Callout>

<Callout kind="alert">

This server uses your Documentation.AI account permissions and can modify live documentation. Connect it only to AI tools and workflows you trust, and use branches when you want review before deployment.

</Callout>

## Setup and authentication

The Authoring MCP Server supports OAuth sign-in and API-key authentication. Use OAuth for interactive MCP clients where you can sign in through your Documentation.AI account. Use API-key authentication for automation and workflows that already store a Documentation.AI `dai_` key.

```text
https://api.documentationai.app/mcp
```

### OAuth sign-in for interactive clients

Connect your MCP client to the server URL, then sign in with OAuth 2.1 through your Documentation.AI account.

<Steps>
  <Step title="Start the connection from your MCP client" icon="terminal">

Open your MCP-compatible client and add the Authoring MCP Server using the URL in the next section.

  </Step>
  <Step title="Sign in with your Documentation.AI account" icon="lock">

Your client discovers the OAuth endpoints and opens a browser window for authentication.

  </Step>
  <Step title="Approve access" icon="check-circle">

After you approve the consent screen, the client receives tokens and completes the connection.

A successful connection shows the server as available in your MCP client.

  </Step>
</Steps>

OAuth handles token management for interactive clients, so you do not need to copy tokens manually.

### API-key authentication for automation

Use API-key authentication when your workflow cannot complete a browser-based OAuth sign-in, such as CI jobs, scheduled scripts, or internal automation.

Configure your workflow to authenticate with a Documentation.AI API key that starts with `dai_`. Pass the key as a Bearer token in the MCP connection's `Authorization` header, such as `Authorization: Bearer dai_example_9xmk7q2r`. The API key uses the same role-based access model as the REST API, so read and write capabilities depend on the key's permissions.

For key creation, roles, and rate limits, see the [REST API overview](/api-reference/overview).

<Callout kind="alert">

Store `dai_` keys in your workflow's secret manager. Do not commit API keys to your documentation repository or MCP client configuration files.

</Callout>

## Configure your MCP client

### Claude Desktop

Add this to your Claude Desktop MCP configuration, `claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "documentationai": {
      "command": "npx",
      "args": [
        "mcp-remote",
        "https://api.documentationai.app/mcp",
        "--timeout",
        "120000"
      ]
    }
  }
}
```

### Claude Code

Add the server from your terminal:

```bash
claude mcp add documentationai \
  --transport http \
  --url https://api.documentationai.app/mcp
```

Or add a `.mcp.json` file to your project root:

```json
{
  "mcpServers": {
    "documentationai": {
      "command": "npx",
      "args": [
        "mcp-remote",
        "https://api.documentationai.app/mcp",
        "--timeout",
        "120000"
      ]
    }
  }
}
```

### Cursor

In Cursor settings, add a new MCP server:

- **Name:** `documentationai`
- **Type:** `command`
- **Command:** `npx mcp-remote https://api.documentationai.app/mcp --timeout 120000`

## Select a project

After authentication, select the documentation project you want to work on before using any documentation tools.

<Steps>
  <Step title="List available projects" icon="folder">

Call `list_projects` to see every organization and documentation project your account can access.

  </Step>
  <Step title="Set the active project" icon="settings">

Call `select_project` with the organization and documentation IDs returned by `list_projects`.

  </Step>
  <Step title="Run read or write operations" icon="edit">

Once a project is selected, use the tools below to inspect files, update content, manage branches, and deploy changes.

A successful selection makes the current project available for subsequent tool calls in the same session.

  </Step>
</Steps>

You can switch projects at any time by calling `select_project` again.

## Tools and capabilities

The server exposes project selection, read, and write tools for repository workflows.

### Project selection

These tools are available immediately after authentication.

<Expandable title="list_projects">

Lists all organizations and documentation projects you have access to, along with your role in each.

**Parameters:** None

**Returns:** A list of organizations, each containing their documentation projects and your role, such as admin, editor, or viewer.

</Expandable>

<Expandable title="select_project">

Sets the active organization and documentation project for the current session.

<ParamField body="organizationId" param-type="string" required="true" show-location="false">
Organization ID from `list_projects`
</ParamField>

<ParamField body="documentationId" param-type="string" required="true" show-location="false">
Documentation ID from `list_projects`
</ParamField>

</Expandable>

### Read tools

Available to all roles, including viewer, editor, and admin. Use these when your workflow needs repository context, source files, or project configuration.

<Expandable title="get_site_config">

Returns the complete `documentation.json`, including navigation, branding, SEO, and integrations.

<ParamField body="branch" param-type="string" required="false" show-location="false">
Branch name. Defaults to the deployment branch.
</ParamField>

</Expandable>

<Expandable title="list_pages">

Lists all files in the documentation repository with their paths and sizes.

<ParamField body="branch" param-type="string" required="false" show-location="false">
Branch name. Defaults to the deployment branch.
</ParamField>

</Expandable>

<Expandable title="get_page">

Returns the full content of a documentation page.

<ParamField body="path" param-type="string" required="true" show-location="false">
File path relative to repository root, for example `getting-started/introduction.mdx`
</ParamField>

<ParamField body="branch" param-type="string" required="false" show-location="false">
Branch name. Defaults to the deployment branch.
</ParamField>

</Expandable>

<Expandable title="search_documentation">

Performs a hybrid search that combines semantic understanding and keyword matching.

<ParamField body="query" param-type="string" required="true" show-location="false">
Search query
</ParamField>

<ParamField body="limit" param-type="integer" required="false" show-location="false">
Maximum results from 1 to 50. Defaults to 10.
</ParamField>

</Expandable>

### Write tools

Require editor or admin role. These tools make the Authoring MCP Server the right choice for authoring, automation, and deployment workflows.

<Expandable title="update_documentation">

Creates, updates, or deletes documentation files in a single Git commit. Supports up to 20 operations per request.

<ParamField body="operations" param-type="array" required="true" show-location="false">
Array of file operations. Each operation includes `type`, `path`, and `content`. `content` is required for create and update operations.
</ParamField>

<ParamField body="commitMessage" param-type="string" required="true" show-location="false">
Git commit message with a maximum length of 500 characters
</ParamField>

<ParamField body="branch" param-type="string" required="false" show-location="false">
Target branch. Defaults to the deployment branch.
</ParamField>

</Expandable>

<Expandable title="create_branch">

Creates a new Git branch.

<ParamField body="branchName" param-type="string" required="true" show-location="false">
Name for the new branch
</ParamField>

<ParamField body="sourceBranch" param-type="string" required="false" show-location="false">
Branch to create from. Defaults to the deployment branch.
</ParamField>

</Expandable>

<Expandable title="list_branches">

Lists all branches in the repository.

**Parameters:** None

</Expandable>

<Expandable title="delete_branch">

Deletes a branch. The deployment branch cannot be deleted.

<ParamField body="branchName" param-type="string" required="true" show-location="false">
Branch name to delete
</ParamField>

</Expandable>

<Expandable title="merge_branches">

Merges one branch into another. Merging into the deployment branch triggers a live deployment.

<ParamField body="base" param-type="string" required="true" show-location="false">
Target branch to merge into
</ParamField>

<ParamField body="head" param-type="string" required="true" show-location="false">
Source branch to merge from
</ParamField>

<ParamField body="commitMessage" param-type="string" required="false" show-location="false">
Merge commit message
</ParamField>

</Expandable>

<Expandable title="list_pull_requests">

Lists pull requests in the repository.

<ParamField body="state" param-type="string" required="false" show-location="false">
Filter by state: `open`, `closed`, or `all`. Defaults to `open`.
</ParamField>

</Expandable>

## Role-based access

Your access level is determined by your role in the selected organization.

| Role | Permissions |
| --- | --- |
| **Viewer** | Read tools only: `get_site_config`, `list_pages`, `get_page`, `search_documentation` |
| **Editor** | Read and write tools, including `update_documentation`, branch operations, merges, and pull requests |
| **Admin** | All tools |

If your team wants an AI client to read documentation without any ability to change it, use the [Docs MCP Server](/ai/docs-mcp-server) instead.
